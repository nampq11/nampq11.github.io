A practical evaluation loop for RAG assistants

25 Jun 2026 · RAG · Evaluation · LLM Ops

Most RAG improvements feel obvious in a demo: add a better embedding model, change chunk size, rerank results, tweak the prompt. The problem is that demos are not distributions. A change that fixes one impressive example can quietly break many boring ones.

The goal is not to build a perfect benchmark. The goal is to make every retrieval change measurable enough that you can say “this helped” or “this is risky”.

The loop

For a production assistant, I like a four-part loop:

  1. Collect real questions. Start with user questions, support-team questions, and synthetic edge cases that represent known failure modes.
  2. Label expected evidence. For each question, keep the document, paragraph, or URL that should support the answer.
  3. Run retrieval separately. Measure whether the right evidence appears in top-k before involving the LLM.
  4. Judge final answers. Check correctness, groundedness, refusal behavior, and tone against the retrieved context.

Key distinction: retrieval quality and answer quality are different bugs. If the right source never appears, prompting will only hide the issue.

Metrics that are simple enough to keep

You do not need twenty metrics. I usually start with these:

  • Recall@k: did the expected source appear in the top k retrieved chunks?
  • MRR: how high was the first useful chunk ranked?
  • Grounded answer rate: did the final answer only use retrieved evidence?
  • Escalation accuracy: did the assistant refuse or escalate when it should?

A tiny evaluation record

Keep the data shape boring. A JSONL file works well because it is easy to append, diff, and review in pull requests.

{
  "id": "faq-042",
  "question": "Can I book a cardiology appointment on Sunday?",
  "expected_sources": ["clinic-hours/cardiology"],
  "must_include": ["available hours", "booking channel"],
  "must_not_include": ["diagnosis", "medical advice"]
}

How I use it before shipping

Every retrieval experiment should produce a small before/after table. If recall improves but grounded answer rate drops, the change is not ready. If average metrics improve while a critical healthcare path regresses, the change is not ready either.

The workflow is intentionally lightweight: collect examples, label sources, run retrieval, judge answers, inspect regressions. Repeat until the assistant becomes boring in the best possible way.
