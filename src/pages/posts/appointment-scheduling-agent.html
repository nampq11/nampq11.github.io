Designing an appointment scheduling agent

11 Jun 2026 · Agents · Automation · Scheduling

An appointment agent is not just a chatbot that can call an API. It is a stateful workflow that touches real calendars, patient preferences, and business constraints. That means the design needs stronger guardrails than a normal Q&A assistant.

The workflow

I prefer to model scheduling as a sequence of explicit states:

  1. Collect: specialty, location, preferred time, patient details, and constraints.
  2. Validate: confirm required fields and check policy constraints.
  3. Search: query available slots with deterministic tools.
  4. Confirm: show options and ask the user to choose.
  5. Commit: book the slot only after confirmation.
  6. Recover: explain conflicts, expired slots, or missing information.

Where the LLM helps

The model is useful for understanding messy language, summarizing options, and deciding the next state. It should not be the source of truth for availability, pricing, or booking success. Those must come from tools.

user message → intent/state parser → validator → scheduling API
                         ↑                 ↓
                  response composer ← tool result

Guardrails worth adding early

  • Idempotency keys for booking calls so retries do not double-book.
  • Structured logs for every tool call, validation failure, and user confirmation.
  • A rollback or cancellation path when downstream systems change state unexpectedly.
  • Human handoff when the user asks for medical advice or the scheduling API is uncertain.

Rule of thumb: let the LLM choose among safe actions. Do not let it create new actions that the product cannot audit.

This is the difference between an impressive demo and a dependable assistant: the demo optimizes for fluency, while the production system optimizes for recoverable state transitions.
